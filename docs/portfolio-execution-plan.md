# Zhang Wei Portfolio Execution Plan

This document extends the 0-12 portfolio plan with release gates that must be applied before implementation, preview deployment, production release, or rollback. It is a planning and workflow document only; it does not implement business code.

## Global Release Gates

- Clean build integrity: release candidates must rebuild from a clean checkout or isolated worktree, proving production media and data are tracked by Git or generated by the deployment pipeline.
- Asset freshness: deployed `index.html` must reference the latest fingerprinted JS/CSS assets, verified with a QA query, cache-disabled browser check, or documented Cloudflare cache strategy.
- Media governance: use `cover` for cards, `source` for details, and `original` only for archive/download contexts. Oversized originals must not render in cards.
- Work status: every work must have `status`; homepage production surfaces allow only `ready`, except intentionally retained `missing-video` entries must show fallback messaging.
- Rights review: every work must have `rightsStatus`; homepage and production allow only `original` or `licensed`.
- Content versioning: release reports must include a content version ID for homepage curation, media library, cases, workflow lab, and gallery-world data, or state the blocker.
- Admin/editor safety: protected content writes require authorization, sanitization, draft/preview/publish separation, schema/content validation, and rollback.

## Phase Additions

### Round 0 - Rules, Skills, Preflight

- Add these blind spot controls to `AGENTS.md` and repo skills.
- Keep changes docs/workflow-only; do not modify runtime code or data schemas in this round.
- Record that `scripts/qa-content.mjs`, clean checkout rebuild, content version IDs, and cache freshness checks are required future gates.

### Round 1 - Production Stop-Loss

- Confirm no homepage item violates `status` or `rightsStatus` once those fields exist.
- Keep missing videos visible with fallback messaging.
- After production deploy, verify asset freshness with `?qa=<commit-or-hash>` and a cache-disabled check.
- If production is touched, smoke-test `/` for nonblank content and no obvious 500.

### Round 2 - Media Inventory And Classification

- Extend media classification to include `cover`, `source`, `original`, `poster`, `status`, and `rightsStatus`.
- Inventory untracked media and block release if production depends on untracked files.
- Plan or add `scripts/qa-content.mjs` to check duplicate covers, missing title/alt/source/poster, missing duration, workflow leakage, invalid status/rights, missing bilingual copy, incomplete cases, and black-card candidates.

### Round 3 - Design System And MediaFrame

- Enforce cover aspect ratios and optimized card assets.
- Keep full-resolution source/original media for detail surfaces only.
- Validate 320px width, 390/768/1440 layouts, and 200% text zoom where layout changes affect public surfaces.

### Round 4 - Video Recovery

- Validate desktop Chrome or Edge plus available mobile Safari and Android Chrome behavior.
- Confirm HLS and MP4 fallback paths and visible fallback messages.
- Confirm closing `VideoLightbox` pauses playback.
- Include video failure state in Playwright QA.

### Rounds 5-7 - Gallery, Workflow Lab, Commercial Cases

- Gallery accepts only single visual images with `status: ready` and `rightsStatus: original|licensed`.
- Workflow boards and ComfyUI/process screenshots remain in Workflow Lab or process details only.
- Commercial cases must include goal, role, tools, deliverables, commercial use, final result, and CTA.
- Cases with incomplete fields, unresolved rights, or mismatched media are preview-only or hidden.

### Round 8 - Hero, Navigation, Immersive Entry

- Define URL behavior for `/`, `?gallery=1`, `?preview=1`, `?lang=en`, and `?qa=<hash>`.
- `?gallery=1` must fall back to the normal homepage if immersive mode fails.
- Preview data must not mutate production data.
- Contact entry points must separate job/recruiting and commercial-collaboration paths.

### Round 9 - I18n And Details

- Default language is Chinese.
- Short-term strategy is `?lang=en` plus localStorage persistence.
- English copy must come from a unified copy map; do not add scattered mixed-language JSX for newly edited translatable surfaces.
- Detail states must refresh without 404 or degrade intentionally to the normal portfolio route.

### Round 10 - 3D Gallery Progressive Enhancement

- Verify ordinary homepage remains the baseline route.
- Validate reduced motion, low-performance fallback, mobile ordinary mode, and `?gallery=1` failure recovery.
- Monitor console errors and resource failures on gallery route.

### Round 11 - Accessibility, Performance, SEO, Contact, Monitoring

- Accessibility QA must check keyboard tab flow, focus visible, Esc close, 320px width, 200% text zoom, reduced motion, and alt text.
- SEO/social QA must check title, description, Open Graph metadata, favicon, canonical, and preview image/copy quality.
- Contact QA must prove email copy feedback, resume download if claimed, and real form backend or visible email fallback.
- Monitoring plan: short-term Playwright checks for resource failures, video playback failures, console errors, and 404s; long-term Cloudflare Analytics, Worker logs, Sentry, or another approved runtime error pipeline.

### Round 12 - Full QA, Preview Acceptance, Production Release

- Run clean checkout rebuild before production.
- Run local and deployed Playwright QA with cache freshness checks.
- Run content QA script when implemented, or record the explicit blocker.
- Release report must include URL, commit hash, deploy ID or Worker version, content version ID, screenshots, QA JSON paths, cache/freshness result, and rollback for both code and content.

## Do Not Implement Yet

- Do not add telemetry SDKs, monitoring dependencies, schema migrations, admin permission changes, or content data shape changes without an explicit implementation task.
- Do not assume `images/curated/` can be ignored; it may contain production media.
- Do not treat current dirty working-tree validation as release-grade proof.
